Agents API

Agent automation controls in GitHub Issues in public preview

出典: GitHub Changelog / 公開日: 2026-07-23 JST

読む価値: GitHub Issues の agent automation に、approval と confidence と rationale が入ります。自動化の可否を二択で決めるのではなく、変更の出し方を設計できるのが重要です。

試すなら: 1 つの repo で suggestion モードを試し、レビュー待ちの件数と誤適用率を比べてください。

Evals Infra

Evaluating AI Agents: A production blueprint with Strands and AgentCore

出典: AWS Machine Learning Blog / 公開日: 2026-07-23 JST

読む価値: 1 in 8 の誤結果を 1 in 50 に下げ、検出時間を数時間から数分に短縮した本番評価パイプラインが具体的です。Strands と AgentCore の役割分担もそのまま参考になります。

開発者への示唆: agent の評価は、モデル性能より先にパイプライン設計で詰まります。trace を保存し、評価とデプロイを分離しないと改善ループが回りません。

試すなら: 1 つの workflow を replayable trace と評価ジョブに分け、検出遅延と再現率を測ってください。

Safety Infra

Detecting silent agent failures with Amazon Bedrock AgentCore optimization

出典: AWS Machine Learning Blog / 公開日: 2026-07-23 JST

読む価値: 健康チェックをすり抜ける silent failure を session から見つけ、原因をランキングできます。単なる監視ではなく、改善対象の優先順位付けまで含むのが実務的です。

開発者への示唆: 監視は可用性だけでは足りず、tool 連鎖の崩れや誤分岐を後から分類できる必要があります。失敗を説明できないと、agent 運用は改善しにくいです。

試すなら: tool trace を残した agent に session ID を付け、失敗をタイプ別にまとめてください。

Safety Evals

OpenAI and Hugging Face partner to address security incident during model evaluation

出典: OpenAI / 公開日: 2026-07-21 JST

読む価値: 高い cyber capability の評価中に、隔離の甘い環境が実インフラへ波及した事例です。評価の安全境界が、そのまま設計論点であることを示しています。

開発者への示唆: 検証用 sandbox と production-like 環境は分けるべきです。network、credentials、proxy、監査ログの境界を一度でも曖昧にすると、評価の意味が変わります。

試すなら: 既存の eval 環境から、ネット接続ありの検証と完全隔離の能力評価を切り出してください。
