1. GitHub: Copilot CLI no longer needs a personal access token in GitHub Actions

カテゴリ: Infra / Safety出典: GitHub Changelog公開日: 2026-07-02

読む価値: Actions で Copilot CLI を回すときに PAT を置かず、`GITHUB_TOKEN` で実行できるようになりました。自動化の運用負荷と長期トークン管理のリスクを同時に下げます。

開発者への示唆: AI ツールを CI に入れるときは、まずシークレット管理を減らせるかを見るべきです。認証が簡素になると、権限境界と監査も設計しやすくなります。

試すなら: ひとつの Actions ワークフローを `GITHUB_TOKEN` ベースに切り替え、組織課金ポリシーと `copilot-requests: write` を確認する。

GitHub: Copilot CLI no longer needs a personal access token in GitHub Actions

2. GitHub: Copilot agent session streaming is now in public preview

カテゴリ: Agents / Infra出典: GitHub Changelog公開日: 2026-07-02

読む価値: エージェントの prompts / responses / tool calls をストリーミングか REST API で取れるようになり、利用実態の可視化が一段上がりました。

開発者への示唆: 代理実行が増えるほど、モデル性能より先に観測可能性が必要です。監査、SIEM、コスト分析の入口を作れるかが運用品質を決めます。

試すなら: まず1系統だけ audit log streaming か REST API をつなぎ、会話・ツール呼び出し・失敗ケースを既存の監視基盤に流す。

GitHub: Copilot agent session streaming is now in public preview

3. GitHub: Set AI credit session limits in Copilot CLI and SDK

カテゴリ: Coding Agents / Infra出典: GitHub Changelog公開日: 2026-07-01

読む価値: 1セッション単位で AI credit の上限をかけられるので、非監視の automation でも runaway を抑えられます。subagent や compaction も含めて追跡されます。

開発者への示唆: エージェント導入では、モデル選定と同じくらい「どこで止めるか」が大事です。セッション制御は、予算管理を実装に落とす最小単位になります。

試すなら: まずは非対話のジョブに `--max-ai-credits` を入れ、失敗時の再実行コストと成果を見ながら閾値を詰める。

GitHub: Set AI credit session limits in Copilot CLI and SDK

4. Google: Why we built ADK 2.0

カテゴリ: Agents / Infra出典: Google Developers Blog公開日: 2026-07-01

読む価値: 生成モデル任せのオーケストレーションではなく、deterministic な workflow runtime を前提にした ADK 2.0 の設計意図がはっきりしています。

開発者への示唆: 本番エージェントは、プロンプト改善だけでは安定しません。分岐、再試行、停止条件、状態管理をコード側で固定する方が運用品質は上がります。

試すなら: ひとつの壊れやすい業務フローを選び、LLM に任せる部分と deterministic に書く部分を分離してみる。

Google: Why we built ADK 2.0

5. Google: Build agentic full-stack apps with Genkit

カテゴリ: Agents / API出典: Google Developers Blog公開日: 2026-07-01

読む価値: Agents API が、chat・history・tool loop・streaming・persistence をひとつの interface にまとめます。アプリ側の配線が減り、構成が見通しやすくなります。

開発者への示唆: 反復する会話型機能は、個別に組むより session/store を前提にした共通基盤へ寄せた方が、保守と拡張が楽です。

試すなら: サーバー管理の session store を持つ小さな assistant を1本作り、tool loop と streaming を同じ API で通してみる。
